# asset-agent — 整合方案

## 当前状态

两个仓库各自验证了一个核心设计点：

| 仓库 | 验证的设计点 | 产物 |
|------|------------|------|
| qtcloud-asset | 版本演化可视化 | 结构树 + 演化图 + app.py |
| qtcloud-agent | 上下文边界管理 | 审计面板 + 边界配置 + 规则引擎设计 |

两者解决同一问题的不同侧面，数据模型互补。

---

## 整合架构

```
                    ┌──────────────┐
                    │   事件总线    │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
      ┌──────────┐ ┌──────────┐ ┌──────────┐
      │ 版本追踪  │ │ 审计日志  │ │ 边界配置  │
      └────┬─────┘ └────┬─────┘ └────┬─────┘
           │             │            │
           ▼             ▼            ▼
      ┌───────────────────────────────────┐
      │         统一渲染层                │
      │  ┌─────────┐ ┌─────────┐ ┌──────┐ │
      │  │ 结构树   │ │ 演化图   │ │审计表│ │
      │  │(分工)   │ │(过程)   │ │(成本)│ │
      │  └─────────┘ └─────────┘ └──────┘ │
      └───────────────────────────────────┘
```

三层架构：

1. **事件总线** — 统一数据源。文件变更、智能体触发、反馈、审计日志都作为事件写入总线。
2. **处理器层** — 三个处理器各自消费事件：版本追踪记录因果链，审计日志记录每次调用的上下文使用，边界配置管理读取权限。
3. **渲染层** — 三个视图消费处理器的输出。共享同一份数据源，各自过滤出需要的节点和边。

## 数据模型统一

两个原型各自定义了图数据模型，核心差异在"边"的语义上。

### 统一方案

```
graphNodes: { id, label, group, parent }
  group ∈ { Dir, File, Agent, Version, Event }

graphEdges: { id, from, to, type, label, events[] }
  type ∈ { write, trigger, feedback, load }
```

新增：
- **Event** 节点类型 — 事件从边属性拆出为独立节点，每条审计日志对应一个 Event 节点
- **load** 边类型 — 智能体→文件，表示本次调用加载了该文件，携带 token 数和引用状态

这样 asset 版的版本演化图和 agent 版的审计表共用同一份 graphData：
- 结构树读 Dir + File + Agent
- 演化图读 Version + trigger/feedback 边
- 审计表读 Event + load 边

## 当前差距

| 模块 | 资产库版 | agent 版 | 整合目标 |
|------|---------|---------|---------|
| 事件总线 | ❌ 无 | 🔧 设计 | 统一事件格式和持久化 |
| 版本追踪 | ✅ 有 | ❌ 无 | 复用资产库版 |
| 审计日志 | ❌ 无 | ✅ 有 | 复用 agent 版 |
| 边界配置 | ❌ 无 | ✅ 有 | 复用 agent 版 |
| 统一渲染 | ❌ 无 | ❌ 无 | 三合一视图 |

---

## 整合策略

1. 统一数据模型（graphData 扩展 Event + load）
2. asset 版的事件写入改成从事件总线消费（当前是 app.py 硬编码 traces）
3. agent 版的审计数据作为 Event 节点写入同一条事件总线
4. 渲染层从两套 HTML 合并为一个三视图看板
