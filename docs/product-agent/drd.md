基于你设计的“探索-整合循环”工作流，从业务人员的视角，以下是支撑整个产品研发体系运转的核心模型、关键事件和状态定义。

—

一、核心业务模型

1. 产品（Product）

业务含义：一个从想法到交付的完整价值载体，是整个工作流的核心管理对象。

关键属性：

· 产品名称（初始为临时名称，最终收敛为正式名称）
· 一句话价值描述（探索期为空，整合期结束后提炼）
· 当前阶段（探索期 / 整合期 / 已发布 / 已归档）
· 原始吐槽来源
· 核心事例摘要
· 关联的代码仓库（原型仓库 + 正式仓库）
· 所属产品线/领域

生命周期：

```
[想法产生] → [探索中] → [事例验证通过] → [重构中] → [已发布] → [迭代中/已归档]
```

—

2. 工作项（WorkItem）

业务含义：看板上流转的任务卡片，代表一个从“要做”到“已完成”的工作单元。工作项是执行层的最小管理单位。

关键属性：

· 工作项类型（吐槽/需求梳理/写事例/写原型/验证/重构/写测试/写文档/发布）
· 所属产品
· 当前状态（对应看板列）
· 负责人
· AI参与标记（AI生成/人工确认/纯人工）
· 关联的临时文档链接
· 关联的代码分支/提交
· 所属阶段（探索期 / 整合期）

工作项类型枚举：

类型 所属阶段 业务含义
吐槽录入 探索期 原始用户声音，未经过滤
需求梳理 探索期 将吐槽结构化，形成临时文档
编写事例 探索期 描述一个具体的、可验证的使用场景
构建原型 探索期 写出能跑通事例的完整应用代码
验证事例 探索期 确认原型是否体现核心价值
代码重构 整合期 套框架、整理结构、消除硬编码
编写测试 整合期 从底层往上覆盖测试
生成文档 整合期 基于验证后的代码倒推文档
提炼产品介绍 整合期 将复杂实现浓缩为一句话价值
发布 整合期 打包正式版本并标记里程碑

—

3. 版本（Version）

业务含义：一次正式交付的快照，是产品对外承诺的载体。

关键属性：

· 版本号（遵循语义化版本）
· 所属产品
· 发布日期
· 制品清单（代码包、文档、测试报告、产品介绍）
· 关联的原型版本（追溯来源）
· 一致性验证状态（文档与代码行为是否一致）
· 一句话产品介绍（此版本的产品介绍）

业务规则：

· 一个产品有多个版本
· 每个版本关联一个或多个通过验证的原型
· 只有通过全部整合期列的工作项，才能触发版本生成

—

4. 资产（Asset）

业务含义：长期保留、值得复用的交付物或知识沉淀物。

资产类型枚举：

资产类型 业务含义 生命周期
原型代码包 探索期产出的可运行事例代码 长期保留（作为历史追溯）
正式代码包 整合期重构后的标准化代码 长期保留（可复用）
测试套件 覆盖底层到高层的全部测试 长期保留（随版本迭代）
精确平台文档 基于代码行为生成的文档 长期保留（与代码版本绑定）
产品介绍文档 人类可读的价值陈述 长期保留（对外展示）
测试报告 某次测试运行的覆盖率/通过率 长期保留（审计追溯）
临时需求文档 BRD/PRD等中间产物 用完即弃，仅存档

业务规则：

· 临时文档标记“生命周期结束”后，不进入资产库，仅在产品描述页折叠存档
· 精确文档与代码行为的一致性需要被持续验证

—

5. 客户反馈（CustomerFeedback）

业务含义：客户通过共创板提交的想法、投票或场景描述，是需求池的外部输入源。

关键属性：

· 反馈类型（吐槽 / 场景补充 / 投票）
· 来源客户标识
· 关联产品
· 关联的探索方向
· 状态（待处理 / 已进入需求池 / 已关闭）

业务规则：

· 反馈被采纳后，自动在工作看板的“需求池”列创建吐槽类工作项
· 反馈与内部工作项保持双向链接，客户可以追踪反馈的处理进展

—

二、关键业务事件

事件是驱动工作项状态流转的触发点，也是工作流可追溯的审计节点。

事件名称 触发时机 业务含义
吐槽提交 内部或客户提交原始问题 新的价值探索起点
需求对齐完成 临时文档编写完毕 团队对问题达成暂时共识
事例编写完成 关键使用场景被描述出来 价值的具象化表达形成
原型构建完成 可运行的原型代码产生 想法首次被代码验证
事例验证通过 原型被确认体现核心价值 探索期终点，允许进入整合
转入整合 原型代码移交整合期 从发散转为收敛的关键决策点
重构完成 代码结构符合工程规范 原型代码转化为平台代码
底层测试通过 单元/模块测试覆盖达标 底层逻辑被锁定
高层测试通过 集成/E2E测试覆盖达标 用户路径被锁定，行为稳固
文档生成完成 精确文档与代码行为一致 知识从代码转化为可读描述
产品介绍提炼完成 一句话说清产品价值 穿越复杂后的简单被沉淀
版本发布 正式版本打包并标记 对外的可信交付完成
反馈采纳 客户反馈进入需求池 外部声音进入内部工作流
一致性验证失败 文档与代码行为不一致 触发回到重构或测试列的反馈回路

—

三、关键业务状态

产品级状态

状态 业务含义
探索中 产品处于探索期，至少有一个进行中的工作项
事例验证通过 原型已验证核心价值，等待进入整合
重构中 产品处于整合期，代码正在规范化
已发布 至少一个版本已正式交付
迭代中 已发布的产品有新的探索方向在进行
已归档 产品不再活跃，进入只读存档

工作项级状态（与看板列对应）

状态 业务含义
待处理 新建工作项，等待开始
进行中 当前正在执行
已完成 此阶段工作完成，等待进入下一列
阻塞 被依赖或问题卡住
已回退 因后续阶段发现问题而退回

版本状态

状态 业务含义
准备中 整合期工作项进行中，版本尚未形成
已发布 正式打包，对外可用
已废弃 此版本因问题被撤回或替代

资产状态

状态 业务含义
有效 资产与当前代码行为一致，可被引用
过期 代码已更新，资产未同步验证
归档 历史资产，仅用于追溯

—

四、模型关系总结

```
产品 (1) ──< (多) 工作项 (探索期 + 整合期)
产品 (1) ──< (多) 版本
版本 (1) ──< (多) 资产
产品 (1) ──< (多) 客户反馈
客户反馈 (多) ──> (1) 工作项 (采纳后创建)
原型代码包 (资产) ──> 正式代码包 (资产) (追溯关系)
```

核心流转：

```
客户反馈 ──采纳──→ 工作项(吐槽录入) ──流转──→ 工作项(验证通过) ──触发──→ 产品状态变更
                                                                        │
                                                         工作项(发布完成) ──生成──→ 版本 + 资产
```

这些模型和事件共同构成了产品研发工作流的业务骨架，也是后续系统设计、数据库建表和API设计的基础输入。
