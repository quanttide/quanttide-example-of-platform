对我从那个代码模块又到思考模块，从思考模块又到文档模块，从文档模块现在又到沟通模块，我感觉我在接近这个根本问题，就是这里面的根本问题是人际沟通的问题，就是人际沟通的共识，这个是根基和基础，就是。人类如果和 AI 在和智能体在语义上不漂移的话，那么就需要对这个对话的沟通结论进行持续的跟踪

那么其实这个就是整个系统的关键了然后也同样是两种设计方法，一种设计方法是专职专能，就是有专门的 AI 去做这个事情。还有一种方法呢，就是就是用一个智能体让他总结

这里我们需要一个沟通状态的管理机制。核心目标是人机沟通产生共识以后存卡片。

之前我在单个系统发现不了问题，是因为必须要把所有的组件都准备好组合起来，才能够适应我想要适应的环境，这个是很关键的一点。所以说之前的做法老是不得劲，就是这个原因
当我们同时从不同的领域去穷举各种智能体的需求，把各种各样的建模全都写上来的时候，你会发现整个建模思路是一环扣着一环。那比如说你像编程这个事情，我们的理解是先写文档后编程，那么就会从文档智能，就会从代码智能体自然而然地去过渡到文档智能体上，然后。写文档的过程中，发现就是意图老是漂移，那么就自然地就过渡到了。沟通的智能体上面，就是人机沟通这个最基本的问题上面


你会发现这个不断的溯源就会找到那个真正让我困扰的问题上面以及真正的解决方案，也就是我一直想做的，去区分沟通状态，然后给他做一个状态机，然后给他的行为做一个合适的建模

我这件事情真正的难点不是单个聊天的状态，而是在沟通的过程之中，特别是探索性这种比较强的过程之中，主题是在不断的变化。

就是之前也不是没有发现这个问题，只是说觉得好像为他专门去建一个状态机，这件事情本身好像还还比较没有必要。现在发现原来这个才是最根本的问题，就是前面说的所谓的这种什么建模啊，什么不同的来源啊，这些都是在寻找共识机制。那么从哪怕从一次聊天之中建立共识机制都没有那么容易

你就比如说，我怎么才能知道这一次对话结束了呢？这种问题就是我怎么才能识别话题的变化，然后多个话题之间要怎么去转移，就它不是单个话题的问题。所以这个建模其实还蛮微妙的，就它不是一个简单的状态
