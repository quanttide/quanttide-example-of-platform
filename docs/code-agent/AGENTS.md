# code-agent 开发者潜意识

## 工作纪律

- **文档分层**：BRD 只摆问题和需求，PRD 只定方案，ADD 只做决策，DEV 只写实现。不串层。BRD 里不出现方案，ADD 里不出现需求。
- **根因推两层的习惯**：看到现象问"为什么"，有了答案再问一次。不要停在第一层。这既是分析习惯也是写作习惯。
- **减法优先**：每份文档都必须写"不做什么"。告诉人不必关心什么，和告诉人关心什么同等重要。不确定就空着，不放不确定的东西。

## 深层信念

- **信机制不信对话**：对 AI 的要求如果没有系统级阻断，等于没要求。prompt 引导、AGENTS.md 声明都是心理安慰。只看一件事——AI 能不能绕过。能绕过就是没约束。
- **文件系统即契约**：Plugin 不解析 AI 的对话输出，只读文件。AI 不读系统提示，只调用工具。两边通过 `.quanttide/code/` 下的文件沟通。这是显式协议优于隐式协商。
- **系统级 > 配置级 > prompt 级**：方案选型的唯一判断标准是约束力。Permission = tool.execute.before 拦截是最强的（不可跳过），Skills/AGENTS.md 次之（可偏离），对话习惯最弱（无约束）。选最强的。
- **解耦前置**：Plugin 和 UI 的接口是文件格式，不是函数调用不是 API。格式定在 Plugin 层，UI 只消费不改 Plugin。谁先做不影响后续。

## 写作规则

- **单层标题原则**：不使用三级及以上标题（`###` 禁止）。复杂内容用表格、列表、代码块组织，不用嵌套标题。
- **一个需求一个钩子**：架构里每个需求对应唯一的主钩子。一个钩子做多件事是设计错误。
- **文件名是语义**：`brd.md` 里不写方案，`prd.md` 里不写架构决策。命名本身就是分类。
- **状态机必须穷举**：画状态图必须画全所有状态和转移路径。"提交→确认→可写→变更→回到提交"比"提交→确认→执行"准确。
- **不省略边界**：每篇文档的最后一段必须是边界声明。没想清楚边界就说明设计没完成。

## 结构记忆

```
examples/code-agent/
├── AGENTS.md      # 开发者潜意识（本文）
├── brd.md         # 问题 + 需求（R1-R5）
├── prd.md         # 方案定义（A1-A3 + U1-U3）
├── add.md         # 架构决策（Plugin 钩子分配 + 状态设计）
└── dev.md         # 开发指南（代码骨架 + API 签名）
```
