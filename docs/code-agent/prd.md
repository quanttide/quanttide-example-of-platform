# code-agent — 意图可见性方案

## 问题定位

> R1-R5 的根因不是"AI 不够听话"，是 **AI 对意图的理解不可见**。
>
> 用户看不到 AI 的理解，所以只能在交付后才发现问题。
> 纠错窗口在最后，不在过程中。这是所有 R1-R5 的共同根因。

## 方案概述

让 AI 的理解**变得可见**——在它动手之前、动手之中、交付之前，持续披露对意图的理解和当前状态。

方案分两层，**必须同时实施**：

- **算法层**：定义 AI 理解的结构化模式、输出规范、检查点协议
- **UI 层**：将算法层的产出可视化为交互界面，让用户在代码之前先看到 AI 的思考

## 算法层

### A1：理解披露协议

AI 接到任务后，先结构化输出以下内容，**写代码前必须完成**：

```
任务目标：<一句话>
交付形态：<代码/文档/配置/调研报告>
验收标准：<可验证的条件列表>
关键假设：<我对任务的隐含理解>
风险点：<我可能理解错的地方>
```

### A2：检查点协议

AI 在执行过程中，遇到以下事件时主动暂停并披露进度：

- 读取完所有相关文件后（写文件前）
- 完成一个子任务单元后
- 切换技术方向前
- 遇到无法自主决策的问题时

披露内容：

```
当前进度：<已完成/总共>
下一步：<将要做什么>
调整记录：<自上一检查点以来的变化>
```

### A3：自检协议

AI 在交付前，对照 A1 的验收标准逐条验证：

```
验收项 1：<标准> → <实际产出> → ✅/❌
验收项 2：<标准> → <实际产出> → ✅/❌
...
未覆盖项：<计划外做了或没做的>
差异说明：<如果有差异，原因>
```

## UI 层

### U1：理解面板

将 A1 的结构化理解渲染为可视区域，支持：

- **卡片式布局**：每个字段独立显示，一眼可扫
- **逐字段确认**：用户可逐条点"确认"或"修改"
- **修改即触发**：用户修改任意字段后，AI 按新版理解重新规划

### U2：进度面板

将 A2 的检查点渲染为时间线视图：

- **阶段标记**：理解→执行→验证，当前阶段高亮
- **检查点列表**：已完成的可展开查看详情
- **暂停指示**：当前在等待用户确认的状态

### U3：审计面板

将整轮交互的关键决策点归集为可追溯的视图：

- 每条记录：时间 + 事件类型（理解/检查点/自检/方向变更）
- 展开可看原始内容
- 支持跨会话串联（输出为文件）

## 协议格式

UI 和算法之间通过结构化数据交换，格式无关（JSON/YAML/Markdown frontmatter），唯一要求：AI 产出的理解和状态**必须能被 UI 独立解析**，不依赖人类阅读。

## 交付条件

1. A1 协议部署后，AI 在写任何业务文件前必须先输出结构化理解
2. U1 能渲染 A1 产出并支持逐项确认/修改
3. A2 协议部署后，AI 在关键事件前自动暂停并输出进度
4. U2 能渲染 A2 产出为时间线视图
5. A3 + U3 支持交付前自检和事后审计
6. 协议格式与 UI 解析器解耦，格式变更只需更新解析层
