问题的本质是：当前AI编程工具的交互范式，是围绕“代码生成”这个行为构建的，而不是围绕“意图形成”这个认知过程构建的。

拆开来看，有三个层面的错位：

1. 时间维度的错位：工具假设意图前置，但实际意图是涌现的。
   现有工具的设计隐含了一个前提——你知道自己想要什么，AI只是帮你更快地实现。但你的工作流是探索型的，意图不是一开始就清晰的，而是在反复试错、阅读代码、丢弃原型的过程中逐渐“蒸馏”出来的。工具不支持这个“意图从模糊到清晰”的渐变过程，它只认最终的那个清晰指令。
2. 信息展示的错位：工具展示执行过程，但你需要的意图状态。
   OpenCode的侧边栏、文件列表、Language Server信息，回答的问题是“AI在干什么”。你真正关心的问题是“我们共同理解了什么”——核心意图是什么、阅读和写作的边界在哪、当前理解有没有漂移。前者是System One的活动日志，后者是System Two的认知看板。这是两种完全不同的信息。
3. 控制权的错位：工具自动管理，但你要求可介入。
   现有的To-Do List、自动补全、代码生成，本质上是AI替你做了决定，然后通知你结果。你要的是“他制定的计划，我能够干预”——AI可以提议，但确认权在你手里。这不是技术问题，是设计哲学的问题：谁才是意图的最终所有者。

—

所以你的困境不是“工具不好用”这个层面。你撞到的是范式墙：从“AI辅助编程”（AI-assisted coding）到“意图驱动的探索工程”（Intent-driven exploration）之间，缺了一整个交互层。 这个层就是你说的System Two——它不写代码，它负责管理“人和AI在认知上有没有对齐”。

这就是问题本质。
