# code-agent — 意图与验收对齐

## 问题

Opencode 经常误解我的意图、上下文和验收标准。

典型表现：

1. **交付形态不符** — 我说"整合"要的是代码，不是文档。我说"重写"要的是不同方案，不是同方案美化。
2. **验收在执行过程中才清晰** — 自己看到产出后才想清楚"这不是我要的"。
3. **短指令 vs 复杂期望** — 一条指令背后有几层隐含假设，没有说出来。

## 对话不够用了

我之前倾向用对话习惯解决对齐问题——指令带验收信号、先对齐再动手、过程检查点。但实践下来，对话约束没有执行力。我要求了它还是按自己默认的理解走，因为我没说不等于它就可以不问。

**习惯不解决结构性矛盾。** 需要刚性约束——不是"建议这么做"，是"不这么做就不能继续"。

## 方向：利用 Opencode 的配置层加约束

不另起炉灶，用 Opencode 现有的配置机制：

### 现有可用的约束面

| 机制 | 约束力 | 说明 |
|------|--------|------|
| AGENTS.md | 中 | AI 在对话开始和文件变更后重读，但违反没有阻断 |
| Skills | 高 | 步骤不可跳过，跳了工作流断掉 |
| opencode.json | 高 | 可配置规则、权限、agent 行为 |
| 自定义 agent | 高 | 可限定工具集和行为模式 |

### 可选方案 A：Skill 约束

定义一个 `code-agent-execute` Skill，强制执行的步骤：

1. 接到指令后先输出"我理解你要做的是 X，产出是 Y 格式"，等确认
2. 确认后按方案执行
3. 交付前输出验收清单，等确认

跳步 = Skill 流程不完整。

### 可选方案 B：AGENTS.md 硬规则

在 AGENTS.md 中声明不可协商的规则：

> 执行任何写文件操作前，必须先输出理解确认。不确认不能继续。

违反规则没有自动阻断，但可以事后追责。

### 可选方案 C：opencode.json agent 配置

将 code-agent 配置为一个独立的 agent，限定其行为模式。

## 不做的方案

- 不另建验收管理系统
- 不加外部工具链
- 不写复杂的规则引擎

## 当前选择

B 最轻量（只加几行 AGENTS.md），A 约束力最强（Skill 不可跳步）。先试哪个？
