# 开发者的业务潜意识

从资产库、agent 两个仓库的 asset-agent 原型对话中推测出的 7 个一致性特征。

**1. 探索先于收敛**

两个仓库做了同一件事的两个侧面，有大量冗余但没急着合并。假设方向先走通再决定，不花时间在整合上直到摸清边界。

**2. 业务语言优先**

同一份内容反复改写——QA 视角 → 构建系统视角 → 业务视角。每次不是润色是重构叙述逻辑。假设一份内容只服务一个读者，通用文档等于给没人看。

**3. 功能先于工程**

永远是功能验证 → 补文档 → 补 README/CONTRIBUTING/ROADMAP → 整理目录。文档和工程规范是给活下来的项目准备的，不是给原型准备的。

**4. 减法与加法冲突**

观念上知道该砍，实际做的时候容易加。9 个 agent 砍到 3 个是减法，三面板 + audit + plan 是加法。两种倾向同时存在，没有统一原则。

**5. 文档厚度 ≠ 质量**

厚文档（brd 185 行、drd 242 行）反而比薄文档更容易缺失 README 和分层目录。该写什么比写多少重要。

**6. 一个人能搞定**

vanilla HTML、Python 单文件、vis-network、file:// 协议、data.js 全局变量。每个选型的共同特征是修改门槛低。原型阶段最大的风险是复杂度，不是能力不足。

**7. 提交即存档**

改什么就提交什么，不攒批、不重写历史。提交的主要价值是存档和推送，不是给后人讲一个工整的故事。
