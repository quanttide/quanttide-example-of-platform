数据流转需求文档（DRD）

1. 文档概述

产品：意图澄清工具
核心功能：在人与AI的探索性对话中，持续捕获、结构化并外化人类意图，形成可编辑、可导出、可作为AI上下文的意图模型。
文档目的：定义意图模型在整个系统生命周期中的数据来源、流转路径、存储形态、变更规则和输出方式。

—

2. 核心数据实体

2.1 意图模型（IntentModel）

意图模型是系统的唯一核心数据实体，由四个字段组成：

字段 类型 业务含义 示例
goal 文本 人类当前要达成的最终状态或核心问题 ”设计一个支持流式响应的多提供商LLM客户端库“
exploration 文本 当前正在尝试的方向、方法或假设 ”采用插件架构实现提供商解耦，使用Python abc模块定义接口“
constraints 文本 已明确的边界条件和限制 ”无第三方依赖；优先支持OpenAI和Anthropic“
state 文本 当前所处阶段及已确认/否定的结论 ”探索阶段；抽象接口已验证可行；流式传输待测试“
updatedAt 时间戳 最后一次修改时间 2025-05-16T14:30:00Z

业务规则：

· 意图模型有且仅有一份实例，不存在多版本并存。
· 任一字段可为空（初始状态），但模型本身始终存在。
· 每次字段变更均更新 updatedAt。

—

3. 数据来源与写入路径

意图模型的数据写入有三种来源，均通过同一写入接口。

3.1 来源一：AI自动识别并提案

触发条件：每轮对话完成后，AI分析用户最新输入。
识别规则：

· AI判断用户输入是否导致意图模型四个维度中任一维度发生结构性变化：
  · 目标：核心问题改变或显著细化
  · 探索：方向切换、假设推翻、新方法提出
  · 约束：新增或移除边界条件
  · 状态：阶段转换（探索→验证、卡住→突破）

数据流转：

1. 用户发送消息 → 消息进入对话上下文
2. 通信层将完整上下文（含当前意图模型）组装为请求，发送至LLM
3. LLM返回回复文本，可能附带结构化意图更新指令
4. 通信层解析回复，分离文本与更新指令
5. 若存在更新指令且通过校验，写入意图模型对应字段
6. UI层收到状态变更事件，刷新意图模型展示，并在对话中插入“意图已更新”提示

写入限制：

· AI仅写入其判定发生变化的字段，未变化字段保持原值
· AI不允许删除字段内容（仅允许修改为空字符串表示明确否定）
· 细节微调（同一方向内的细化描述）不触发写入

—

3.2 来源二：人类手动编辑

触发条件：用户在意图模型面板（桌面右栏/移动顶部）中直接编辑任意字段。

数据流转：

1. 用户点击意图模型字段进入编辑态
2. 用户修改内容后失焦或确认
3. UI层直接调用写入接口，更新对应字段
4. 状态层发布变更事件
5. 通信层在下一次LLM请求中自动携带最新意图模型

写入规则：

· 人类编辑具有最高优先级，直接覆盖当前字段值
· 无确认弹窗、无二次校验
· 若与AI同时修改同一字段（极低概率），以最后一次写入为准

—

3.3 来源三：对话指令

触发条件：用户在对话中显式发出意图修改指令（如“把目标改成X”）。

数据流转：

1. 用户输入指令 → 进入对话上下文
2. LLM识别此指令属于意图修改，在回复中附带结构化更新指令
3. 后续流程与来源一相同

业务说明：此来源本质上是来源一的特例（用户显式触发），区别在于LLM的识别逻辑——对话指令的意图修改确定性更高，无需额外判断结构性变化。

—

4. 数据读取与使用路径

意图模型有四种读取场景：

场景 读取方 读取时机 读取内容
AI上下文注入 通信层 每次发送LLM请求前 完整意图模型，格式化为系统提示
界面展示 UI层 初始加载 + 收到变更事件后 完整意图模型，按字段分区域渲染
BRD导出 导出函数 用户点击导出按钮时 完整意图模型，按字段顺序拼接为Markdown
新对话初始化 通信层 用户开启新会话时 完整意图模型，作为初始系统提示

上下文注入格式（AI读取用）：

```
当前意图模型：
## 目标
{goal}
## 当前探索
{exploration}
## 约束
{constraints}
## 状态
{state}
```

BRD导出格式（人类/其他AI读取用）：

```
# 意图文档
生成时间：{updatedAt}

## 目标
{goal}

## 当前探索
{exploration}

## 约束
{constraints}

## 状态
{state}
```

5. 数据生命周期

5.1 创建

意图模型在会话开始时自动创建，所有字段初始为空字符串。

5.2 更新

贯穿整个会话周期。每次更新覆盖对应字段值，无历史版本保留。
更新频率由探索节奏决定，结构性变化通常发生在：

· 方向转折时（如放弃一个方案尝试另一个）
· 假设被验证或推翻时
· 阶段性总结时

5.3 持久化

· 当前会话：意图模型保留在内存状态中
· 跨会话：导出为BRD文档实现持久化；新会话可通过粘贴BRD内容作为初始上下文，由LLM重新解析填充意图模型

5.4 销毁

会话结束时，内存中的意图模型自动释放。已导出BRD不受影响。

6. 数据流转全景图

```
用户输入 ──→ 对话上下文 ──→ LLM请求（含意图模型）
                                │
                                ▼
                          LLM处理
                                │
                    ┌───────────┴───────────┐
                    ▼                       ▼
              普通文本回复            意图更新指令
                    │                  [INTENT_UPDATE]
                    │                       │
                    │                       ▼
                    │              解析并写入意图模型
                    │                       │
                    └───────────┬───────────┘
                                ▼
                          UI层获取回复
                                │
                ┌───────────────┼───────────────┐
                ▼               ▼               ▼
          对话展示更新   意图面板刷新     “意图已更新”提示

人类手动编辑 ──→ 直接写入意图模型 ──→ UI刷新 ──→ 下次请求携带
导出按钮     ──→ 读取意图模型 ──→ 拼接Markdown ──→ 剪贴板/下载
```

7. 数据质量要求

维度 要求
准确性 AI更新后人类可在任意时刻审视修正，人类编辑具有最终决定权
及时性 AI在每轮对话后即时判断并更新，无延迟
完整性 四个字段覆盖意图形成的主要维度，字段内不限制内容长度
一致性 意图模型是唯一真相源，AI上下文、UI展示、导出均读取同一份数据
隐私性 意图模型仅存在于本地会话，不上传至服务端（取决于部署架构）

—

8. 边界与约束

· 不与代码产物耦合：意图模型仅管理认知状态，不管理文件、版本、构建等工程产物
· 不与任务管理耦合：意图模型不拆解为可执行任务或子任务，保持为认知快照
· 不保留历史版本：每次修改覆盖前值，无版本回溯。如需追溯，依赖对话历史或导出文档
· AI更新为提案性质：通过人类随时可编辑的特性，AI的意图更新本质上是“待人类默认确认的提案”
